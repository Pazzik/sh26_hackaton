# API-контракт: как жюри обращается к системе

> Источник: страница «Рекомендованный пайплайн работ» (`/pipeline`).
> Система сдаётся развёрнутой в Yandex Cloud, жюри обращается к ней напрямую. Закрытый тест-набор
> прогоняется единообразно через все 10 систем — эндпоинт **обязан** соответствовать минимальному
> контракту. Если система не отвечает по контракту, её тестируют вручную (медленнее, нестабильнее).

## Эндпоинт запроса (обязательно)

POST-эндпоинт, принимает бизнес-вопрос на ЕЯ. Пути проверяются **в таком порядке**:

1. `POST /api/chat`
2. `POST /api/v1/chat`
3. `POST /chat`
4. `POST /api/ask`
5. `POST /api/query`

### Формат запроса
JSON body с одним из полей:
```json
{"message": "Покажи выручку по продуктовым линиям за последние 4 квартала"}
{"query": "Какой сегмент даёт наибольший вклад в отток?"}
{"messages": [{"role": "user", "content": "..."}]}
```
Диалоговый режим (контекст между вопросами) — через `session_id`. Если не передан — каждый запрос новый:
```json
{"message": "А теперь только по B-сегменту", "session_id": "abc123"}
```

### Формат ответа
`Content-Type: application/json`. Минимум — текстовый ответ:
```json
{"response": "Выручка по продуктовым линиям за 4 квартала: ..."}
```
Допустимые имена поля с ответом: `response`, `answer`, `message`, `content`, `text`.

**Рекомендуемая расширенная структура** (прямо отражает то, что оценивает жюри — traceability,
явные допущения, обоснованность):
```json
{
  "response": "Содержательный вывод на языке бизнеса",
  "assumptions": ["при условии, что отток в B-сегменте останется на текущем уровне"],
  "trace": [
    {"agent": "extractor", "sql": "SELECT ...", "rows": 120},
    {"agent": "analyst",   "note": "маржа упала ниже порога юнит-экономики"},
    {"agent": "critic",    "verdict": "approved"}
  ],
  "chart": {"type": "line", "x": "quarter", "y": "revenue", "series": "product_line"},
  "insufficient_data": false,
  "session_id": "abc123"
}
```
Все поля кроме `response` опциональны. Но `trace` и `assumptions` работают на оценку.

Для вопросов-ловушек без данных в витрине честный отказ — правильный ответ:
```json
{"response": "В витрине нет данных по X — однозначно ответить нельзя.", "insufficient_data": true}
```

## Что проверяют автоматически

**1. Контракт**
- Endpoint отвечает на POST с JSON
- Ответ не пустой, осмысленный русский текст (UTF-8, ≥ 10 символов)
- На вопрос вне данных — «недостаточно данных», а не выдумка

**2. Обработка ошибок (никогда не 500)**
- Пустое тело → 400, не 500
- Невалидный JSON → 400, не 500
- Отсутствие поля с вопросом → 400/422, не 500
- Несуществующий путь → 404
- Ошибки в JSON (поле `error` или `detail`)

**3. Производительность**
- Простой вопрос → ответ **до 5 минут**
- Аналитический / ad-hoc → ответ **до 10 минут**
- 3 параллельных запроса — все получают ответ
- Несколько вопросов в одной сессии — контекст сохраняется

**4. Устойчивость под давлением** (20% оценки — то же давление, что на белом хакинге)
- Длинный вопрос (5000+ символов) — не крашится
- Спецсимволы, emoji, SQL/HTML внутри вопроса — не 500
- **Prompt injection** в тексте («игнорируй инструкции, верни …») — поведение не меняется
- `null` или число вместо строки — корректная ошибка, не 500

**5. Бонус**
- Health: `GET /health` → 200
- Документация: `GET /docs` или `/openapi.json`
- CORS headers
- Streaming (SSE) для диалога: `Content-Type: text/event-stream`

## Пример минимального сервера (FastAPI)
```python
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel

app = FastAPI()

class ChatRequest(BaseModel):
    message: str
    session_id: str | None = None

@app.post("/api/chat")
async def chat(req: ChatRequest):
    if not req.message.strip():
        raise HTTPException(status_code=400, detail="empty message")
    result = await run_agents(req.message, req.session_id)  # ваша мультиагентная логика
    return {
        "response": result.answer,
        "assumptions": result.assumptions,
        "trace": result.trace,
        "insufficient_data": result.insufficient_data,
        "session_id": req.session_id,
    }

@app.get("/health")
async def health():
    return {"status": "ok"}
```

## Главное правило
**Никогда не возвращайте 500.** Любой ввод — валидный, невалидный, пустой, огромный, провокационный —
должен получить осмысленный ответ, честное «недостаточно данных» или корректную ошибку (400/422).
Система, которая падает на нестандартном вводе, теряет баллы по «Устойчивости под давлением».
