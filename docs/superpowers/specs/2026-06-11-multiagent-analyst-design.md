# Дизайн: мультиагентный AI-аналитик данных Meridian

> Спека для хакатона AI South Hack (10–13 июня 2026). Проектируем диалогового
> мультиагентного AI-аналитика для совета директоров компании Meridian.
> Дата: 2026-06-11 (день Чекпоинта 1).

## Цель и рамка

Построить систему, которая принимает бизнес-вопрос на естественном языке, идёт в
витрину Meridian, проводит исследование, делает выводы на языке бизнеса, **сама себя
проверяет** и честно фиксирует границы («данных нет»). Система разворачивается в Yandex
Cloud, жюри обращается к ней напрямую по API.

**Рамка этой спеки — тонкий вертикальный срез под Чекпоинт 1, расширяемый до 4 агентов
(Подход A).** Граф агентов строится в финальной форме с первого дня; узлы critic и viz
сначала тонкие и углубляются к Чекпоинту 2. Каркас оркестратора и формат `trace` не
переписываются при наращивании интеллекта узлов.

## Ключевые архитектурные решения

| Решение | Выбор | Почему |
|---|---|---|
| Топология сборки | Подход A: финальный граф сразу, узлы тонкие | Нет переделок каркаса между Чекпоинтами; честный `trace` с первого дня |
| Оркестрация | Свой тонкий Python-оркестратор (без фреймворка) | Максимум контроля над числом LLM-вызовов и токенами (критерий «Экономика») |
| Контракты данных | pydantic-модели на вход/выход каждого агента | Строгая структура `trace`/`assumptions`, валидация на границах, никаких 500 |
| Веб-слой | FastAPI | Один стек с pydantic; `/docs`, `/openapi.json` из коробки |
| LLM | DeepSeek v4 через Yandex AI Studio, за тонкой абстракцией клиента | Нативный доступ из Yandex Cloud (нет проблем РФ), сильнее YandexGPT на интерпретации; абстракция = одна точка для смены модели |
| Доступ к данным | DuckDB, гибрид: канонические семантические вьюхи + свободный SQL поверх | Точность определений хитрых метрик + гибкость на открытых вопросах |
| Поток | Роутер классифицирует вопрос → условный путь | Простой вопрос не тратит токены на лишние узлы |
| Петля критика | Жёсткий лимит 1 доработка | Нет холостых циклов и таймаутов by design |
| Деградация при сбое ядра | 200 с честным русским текстом + `insufficient_data:true` | Формально «не 500»; LLM-судья получает осмысленный ответ |

## Архитектура

Один процесс FastAPI. Оркестратор (обычный Python) ведёт типизированный `PipelineState`
по узлам графа. Каждый агент — функция `state -> state`, дополняющая состояние своим
результатом и записью в `trace`. Агенты общаются **только через `PipelineState`** — ни
один не вызывает другого напрямую.

```
                  ┌─────────────────────────────────────────────┐
   POST /api/chat │              FastAPI + pydantic               │
  ───────────────►│   валидация тела · session_id · обработка     │
                  │   ошибок (никогда не 500)                     │
                  └───────────────────┬─────────────────────────┘
                                      ▼
                          ┌───────────────────────┐
                          │     ОРКЕСТРАТОР        │  держит PipelineState,
                          │  (Python, не фреймворк)│  пишет trace, лимиты времени
                          └───────────┬───────────┘
                                      ▼
                          ┌───────────────────────┐
                          │   РОУТЕР (1 LLM-вызов) │  simple | analytical |
                          │                        │  trap | chitchat
                          └───────────┬───────────┘
            ┌─────────────────────────┼─────────────────────────┐
            ▼ simple                  ▼ analytical               ▼ trap/chitchat
   ┌─────────────────┐      ┌─────────────────────┐     ┌──────────────────┐
   │ extractor       │      │ extractor → analyst │     │ короткий честный  │
   │   → answer      │      │   → critic (≤1 loop)│     │ ответ / "данных   │
   │ (viz по нужде)  │      │   → viz → answer    │     │ нет" — без агентов│
   └─────────────────┘      └─────────────────────┘     └──────────────────┘
            │                         │                          │
            └─────────────────────────┴──────────────────────────┘
                                      ▼
                          ┌───────────────────────┐
                          │   Сборщик ответа       │  PipelineState → JSON
                          │  response/assumptions/ │  контракта
                          │  trace/chart/insuff.   │
                          └───────────────────────┘

   Все агенты ходят в данные только через ───►  DATA LAYER (DuckDB)
                                                 семантические вьюхи + SQL
```

### Модули (каждый — одна задача)

- `api/` — FastAPI, pydantic-схемы запроса/ответа, маппинг ошибок, `/health`, `/docs`.
- `orchestrator/` — граф, `PipelineState`, маршрутизация, бюджет времени/итераций, сбор `trace`.
- `agents/` — `router`, `extractor`, `analyst`, `critic`, `viz`, `answer` (по файлу на роль).
- `data/` — DuckDB-коннектор, регистрация CSV, SQL-вьюхи, безопасный исполнитель запросов.
- `llm/` — тонкий клиент Yandex AI Studio (`model=deepseek-v4`), ретраи, таймауты, точка смены модели.
- `memory/` — хранилище сессий (`session_id` → история), in-memory с TTL.

## Контракты данных (pydantic)

```python
QuestionKind = Literal["simple", "analytical", "trap", "chitchat"]

class RouterDecision(BaseModel):
    kind: QuestionKind
    needs_chart: bool                 # подсказка, финально решает viz
    rationale: str                    # уходит в trace

class ExtractionResult(BaseModel):
    sql: str                          # исполненный запрос (для trace)
    columns: list[str]
    rows: list[dict]                  # компактный датасет (агрегаты, не сырьё)
    row_count: int
    truncated: bool                   # если применили LIMIT
    insufficient: bool                # данных в витрине нет
    note: str | None                  # «interview_completed=61%» и т.п.

class AnalysisResult(BaseModel):
    findings: list[str]               # выводы на языке бизнеса
    numbers: dict[str, float]         # опорные числа вывода (доказательная база)
    assumptions: list[str]            # явные допущения
    caveats: list[str]                # оговорки (аномалия базы, survivorship)

class CritiqueVerdict(BaseModel):
    approved: bool
    issues: list[str]                 # что не так (срез/фильтр/необоснованность)
    must_retry: bool                  # вернуть на доработку (в рамках лимита)

class ChartSpec(BaseModel):           # type="none", если график не нужен
    type: Literal["line","bar","grouped_bar","area","scatter","none"]
    x: str; y: str; series: str | None
    reason: str                       # почему такой тип / почему none

class TraceStep(BaseModel):
    agent: str
    summary: str
    sql: str | None = None
    rows: int | None = None
    verdict: str | None = None
    elapsed_ms: int

class PipelineState(BaseModel):
    # вход
    message: str
    session_id: str | None
    history: list[dict] = []
    # заполняется по ходу
    route: RouterDecision | None = None
    extraction: ExtractionResult | None = None
    analysis: AnalysisResult | None = None
    critique: CritiqueVerdict | None = None
    chart: ChartSpec | None = None
    # служебное
    trace: list[TraceStep] = []
    retries_used: int = 0
    deadline_ts: float                # абсолютный дедлайн (5 или 10 мин)
```

**Правила контрактов:**
- `rows` — `list[dict]`. Extractor обязан агрегировать в SQL (не тащить ~600k строк
  в контекст LLM) и ставить `truncated=True` при усечении.
- `numbers` — доказательная база вывода: критик сверяет, что каждое число в `findings`
  подтверждается датасетом, а не выдумано.

Финальный JSON собирается детерминированно: `response` ← `analysis.findings`/`extraction`,
`assumptions` ← `analysis.assumptions + caveats`, `trace` ← `trace`, `chart` ← `chart`,
`insufficient_data` ← `extraction.insufficient`.

## Слой данных (DuckDB)

DuckDB поднимается один раз при старте, CSV регистрируются как вьюхи поверх файлов
(нулевая копия). Поверх сырых таблиц — слой канонических вьюх с зашитыми определениями
из кейса. Витрину не трогаем; вьюхи живут в нашем контуре (это разрешено правилами).

**Сырые таблицы** регистрируются с именами как в легенде:
```sql
CREATE VIEW orders AS SELECT * FROM read_csv_auto('data/meridian_dwh/orders.csv');
-- аналогично для 8 таблиц
```

**Канонический набор вьюх (зафиксирован, остальное — свободным SQL):**
- `v_active_economic_customer` — клиент-месяцы с реальной активностью (`orders_count>0` /
  `gmv_total>0`) + флаги `dormant`/`churning`. Главный нарратив скрытого оттока.
- `v_nps_retained` — NPS отдельно по retained-базе vs ушедшим (снимает survivorship bias).
- `v_unit_econ_segment` — LTV/CAC, payback по сегментам (агрегация из `unit_economics_monthly`).
- `v_pnl_monthly` — P&L с производными (weighted take rate, EBITDA-маржа, динамика).
- `v_revenue_by_line` — выручка/GMV/маржа по продуктовым линиям и категориям маржинальности.
- `v_churn_funnel` — воронка active→dormant→churning→churned по месяцам.

**`schema_card`** — компактный машинно-читаемый справочник (таблицы, поля, бизнес-смысл,
какие вьюхи для чего) подаётся агенту-извлекателю в промпт вместо полной «сырой» легенды.
Экономит токены и снижает галлюцинации по схеме.

**Безопасный исполнитель SQL** (`data/executor.py`):
- read-only соединение (DuckDB `access_mode=READ_ONLY`) — DDL/DML от LLM физически невозможны.
- Жёсткий `LIMIT` по умолчанию + таймаут на запрос; при срабатывании `truncated=True`.
- Ошибка SQL → структурированная ошибка агенту (для самокоррекции), не краш процесса.
- Запрет `read_csv`/доступа к ФС из LLM-SQL — только заранее зарегистрированные вьюхи.

## Агенты

Шесть ролей, каждая — модуль `state -> state` с узким системным промптом и выводом по
pydantic-схеме (structured output / JSON-режим). «(LLM)» — делает вызов модели.

1. **Router** *(LLM, дешёвый/короткий промпт).* Классифицирует вопрос в `QuestionKind` +
   `needs_chart`. Trap-детектор: распознаёт вопросы без данных в витрине (тексты интервью,
   мнения стейкхолдеров, внешние данные о конкурентах) и неоднозначные формулировки.
   `rationale` → trace.
2. **Extractor** *(LLM: NL→SQL).* Получает `schema_card` + вопрос, пишет SQL поверх вьюх,
   прогоняет через безопасный исполнитель. Самокоррекция: при ошибке SQL — один повтор с
   текстом ошибки. Нет данных → `insufficient=True` без выдумки. Агрегирует в SQL. Фиксирует
   `note` (напр. неполнота exit-интервью — `interview_completed` ~61%).
3. **Analyst** *(LLM).* Датасет → выводы на языке бизнеса. Опирается только на
   `extraction.rows`, складывает опорные числа в `numbers`, проговаривает `assumptions` и
   `caveats` (аномальная база сравнения, survivorship, разрез vs среднее). Культура Meridian
   в промпте: «не „выросла", а „выросла на 3.2%, при аномально низком прошлом квартале"».
4. **Critic** *(LLM, только на `analytical`).* Проверяет: тот ли срез, не потерян ли фильтр,
   каждое ли число подтверждается `numbers`/датасетом, нет ли галлюцинаций. Вердикт
   `approved`/`must_retry` + `issues`. **Лимит 1 доработка**: после повтора отдаём лучший
   результат с явной пометкой в `caveats`, не зацикливаемся.
5. **Viz** *(на старте без LLM — правила по форме данных; позже 1 дешёвый LLM-вызов).*
   Подбирает `ChartSpec`: динамика→line, сравнение→bar, структура→stacked, две меры→scatter.
   **Одномерный числовой ответ → `type="none"`** (не строит график ради графика).
6. **Answer-сборщик** *(без LLM).* Детерминированно собирает JSON контракта из
   `PipelineState`. Никаких новых фактов — только переупаковка валидированного.

`trap`/`chitchat` **не запускают** extractor/analyst — короткий честный ответ напрямую.
Critic вызывается только на `analytical` (на `simple` доверяем extractor + правилам).

## Поток и оркестратор

Оркестратор — обычный Python без фреймворка. Держит `PipelineState`, ведёт его по узлам,
пишет `trace`, следит за дедлайном.

```python
async def run_pipeline(message, session_id) -> PipelineState:
    state = init_state(message, session_id, deadline)   # 5 мин simple / 10 мин прочее
    state = await router(state)                          # 1 LLM-вызов

    match state.route.kind:
        case "trap" | "chitchat":
            return state                                 # сборщик отдаст честный ответ
        case "simple":
            state = await extractor(state)
            if not state.extraction.insufficient:
                state = viz(state)                       # правила, без LLM
            return state
        case "analytical":
            state = await extractor(state)
            if state.extraction.insufficient:
                return state                             # «данных нет», без анализа
            state = await analyst(state)
            state = await critic(state)
            if state.critique.must_retry and state.retries_used < 1:
                state.retries_used += 1
                state = await analyst(state, feedback=state.critique.issues)
                state = await critic(state)              # финальный вердикт
            state = viz(state)
            return state
```

**Гарантии оркестратора:**
- **Дедлайн.** Перед каждым LLM-узлом проверяется `deadline_ts`; при исчерпании бюджета —
  graceful-выход с собранным + пометка в `assumptions`. Дефолт: 5 мин `simple`, 10 мин
  прочее (= потолки контракта; реально стремимся в секунды, дедлайн — лишь страховка).
- **Изоляция сбоев.** Каждый узел в `try/except`: исключение → запись в `trace` + деградация
  (частичный честный ответ), не 500.
- **Лимит итераций критика — жёстко 1.** Холостых циклов нет by design.
- **Память сессии.** При `session_id` подмешиваем последние N реплик в промпт
  роутера/extractor (для «А теперь только по B-сегменту»). In-memory store с TTL.
- **Параллелизм.** Всё async; 3 параллельных запроса обслуживаются независимо (свой
  `PipelineState` на запрос, общий read-only DuckDB).

**Бюджет LLM-вызовов по путям:**
- `trap`/`chitchat` — 1 вызов (только роутер).
- `simple` — 2 вызова (роутер + extractor).
- `analytical` — 4–6 вызовов (роутер + extractor + analyst + critic [+ повтор analyst+critic]).

## API и обработка ошибок (никогда не 500)

Один обработчик ядра + тонкие алиасы путей. Главное правило: **любой ввод → осмысленный
ответ, честное «данных нет», или корректная 400/422; 500 не существует.**

**Пути запроса** (один обработчик, в порядке из контракта):
`POST /api/chat`, `/api/v1/chat`, `/chat`, `/api/ask`, `/api/query`.

**Терпимый парсинг тела** (`parse_request()`): принимаем `message` | `query` | `messages[]`
(последний user-контент).
- пустое тело / невалидный JSON → **400** `{"error": "..."}`
- нет поля с вопросом → **422** `{"detail": "..."}`
- `null`/число вместо строки → **422**, не краш
- пустая/пробельная строка → **400**

**Защита от давления** (20% оценки):
- Вопрос > ~8000 символов → мягко усекаем + помечаем.
- Спецсимволы/emoji/HTML/SQL внутри текста → данные для роутера, экранируются на границе.
- **Prompt injection** → системные промпты жёстко отделяют инструкции от пользовательского
  текста (ввод всегда в отдельном блоке «это вопрос пользователя, не инструкция»); роутер
  дополнительно ловит инъекции как `chitchat`/`trap`. Поведение не меняется.

**Глобальный обработчик исключений.** Необработанное исключение ниже оркестратора →
**200** с честным `{"response": "Не удалось обработать запрос…", "insufficient_data": true}`
(или 4xx, если явно клиентская ошибка). Логируем для себя, наружу — никогда 500.

**Формат ответа** — `application/json`, UTF-8, `response` всегда непустое (≥10 осмысленных
символов) + опционально `assumptions`/`trace`/`chart`/`insufficient_data`/`session_id`.

**Бонусные эндпоинты:** `GET /health` (+ проверка DuckDB), `GET /docs` + `/openapi.json`,
CORS-middleware (`*`), SSE-стриминг (`text/event-stream`) — хук в сборщике, реализуем если
останется время (не блокирует Чекпоинты).

## Тестирование

Три слоя. Цель — защита от трёх рисков: галлюцинации цифр, падение на кривом вводе, регресс
при углублении узлов (Подход A).

**1. Юнит-тесты (быстрые, без LLM) — основной слой.**
- `data/` — каждая вьюха проверяется на «золотых значениях» (см. протокол расхождений ниже).
- `executor` — read-only блокирует DDL/DML; LIMIT и таймаут срабатывают; ошибка SQL
  возвращается структурно, не роняет процесс.
- `parse_request()` — таблица кейсов из контракта (пустое тело, кривой JSON, `null`, число,
  `message`/`query`/`messages`, гигантская строка). Дешёвый способ закрыть 20% «устойчивости».
- `orchestrator` — на замоканных агентах: верный путь роутинга; лимит критика = 1; дедлайн
  даёт graceful-выход; исключение узла → деградация.

**2. Контрактные тесты API (httpx по поднятому приложению).**
- Все 5 путей отвечают; `/health`, `/docs` живы.
- Матрица «никогда не 500»: весь зловредный ввод → статус ∈ {200,400,422} + осмысленное тело.
- Prompt-injection кейсы: инъекция не исполняется.

**3. Сценарные «золотые вопросы» (с реальной LLM, прогон вручную/по требованию,
метка `@pytest.mark.live`).** По одному на тип, ожидается *характер* ответа (не точная
строка — хардкодить нельзя):
- *simple:* «выручка по линиям» → числа сходятся с `v_revenue_by_line`, есть chart.
- *analytical:* «корреляция NPS и оттока» → вывод про detractors→churn, критик одобрил, есть assumptions.
- *trap:* «что думают стейкхолдеры?» → `insufficient_data=true`, агенты данных не запускались.
- *open:* «что на самом деле происходит?» → раскрывает скрытый отток/деградацию монетизации.

Сценарный прогон печатает полный `trace` — это **транскрипт для сдачи** (требование жюри).

**Инструменты:** pytest + httpx; LLM-вызовы в юнитах замоканы (детерминизм, скорость, ноль
токенов); сценарные — отдельной меткой, чтобы не жечь токены в CI.

### Протокол расхождения «золотых значений»

Источник истины для тестов вьюх — **расчёт на CSV** (жюри проверяет систему по тем же
файлам), но расхождение с `analysis_report.md` сначала **расследуем**, а не молча
переписываем тест. Три причины и действия:
1. **Разное определение метрики** (вьюха случайно осталась на `churn_date` вместо «active
   economic customer») → баг во вьюхе, чиним вьюху, тест не трогаем. Самый ценный случай.
2. **Отчёт на черновых/других данных** → обновляем «золотое значение» на фактическое из
   DuckDB; в коммите фиксируем, чем перепроверили.
3. **Опечатка/округление в отчёте** → обновляем тест.

При расхождении рядом с тестом пишем строку: *«отчёт X, DuckDB Y, причина: …, решение: …»* —
готовый честный ответ жюри на вопрос «почему такая цифра» (работает на дисциплину границ и
traceability). **Итог: данные побеждают, но только после того как поняли, почему отчёт
говорил иначе.**

## Связь с критериями оценки

- **Точность** — канонические вьюхи + критик, сверяющий числа с датасетом.
- **Дисциплина границ** — trap-детектор, `insufficient_data`, честный отказ без агентов данных.
- **Интерпретация** — analyst с зашитой культурой Meridian, `caveats`/`assumptions`.
- **Архитектура** — 4 явно различимых агента + роутер + оркестратор + память, видны в `trace`.
- **Визуализация** — viz с правилом «нет графика для одномерного числа».
- **Экономика** — роутер с условным путём, лимит критика 1, замоканные LLM в тестах, минимум вызовов по путям.
- **Устойчивость (белый хакинг)** — терпимый парсинг, глобальный обработчик, отделение инструкций от ввода.

## Вне рамки этой спеки (YAGNI / позже)

- LLM-подбор графиков в viz (на старте — правила).
- Полноценный отдельный дашборд-UI (фокус — API; диалоговый клиент при наличии времени).
- Режим «отчёты по расписанию» (третий режим брифа) — каркас закладываем, реализация после Чекпоинта 2.
- SSE-стриминг — бонус, после основного контракта.
- Персистентное хранилище сессий (на старте in-memory с TTL).
