# Dev-просмотрщик с реальным графиком

**Дата:** 2026-06-11
**Статус:** дизайн утверждён, готов к плану реализации

## Проблема

Сейчас результаты системы видно только через `curl`/Swagger. В ответе `/api/chat`
есть `chart` — но это лишь **спека** (`type`, `x`, `y`, `series`, `reason`), без
данных ряда: сами строки (`ex.rows`) в JSON клиенту не отдаются. Нужен лёгкий
dev-инструмент, чтобы быстро гонять вопросы и видеть `response` / `assumptions` /
`trace` / **реальный график** глазами, а не разбирать JSON.

Побочная ценность: реальный рендеринг графика делает наблюдаемым известный баг
агента визуализации — ось Y маппится на `cols[1]` вслепую, поэтому при SELECT с
тремя колонками `(измерение, измерение2, метрика)` на Y попадает категориальная
колонка вместо числовой (см. `viz.py:25`). На графике это видно сразу.

## Назначение и объём

Инструмент **для разработки/тестирования**, не demo-grade UI. Приоритет —
удобство, не красота. Тем не менее изменение backend (отдавать данные ряда)
аддитивно и переиспользуется на будущем демо для жюри.

Вне объёма: правка самого бага осей, изменение логики агентов/роутинга,
аутентификация, сборка фронтенда (bundler), адаптив/стили под продакшн.

## Подход

Выбран **Подход 1**: данные ряда едут внутри `ChartSpec.data` — аддитивное
расширение боевого контракта `/api/chat`. Dev-страница тестирует ровно тот путь,
который видит жюри (без расходимости debug-эндпоинта). Ограничение по числу точек
защищает ответ от раздувания.

Отвергнуто:
- отдельный `POST /api/debug` — раздваивает код, dev тестирует не боевой путь;
- top-level `rows` в ответе — раздувает ответ жюри (до `SQL_ROW_LIMIT`=5000 строк)
  и отдаёт лишнее.

## Изменения backend

### `app/contracts.py`
В `ChartSpec` добавить опциональное поле:
```python
data: list[dict] = []
```
По умолчанию пусто — обратно совместимо.

### `app/config.py`
Новый параметр:
```python
max_chart_points: int = int(os.getenv("MAX_CHART_POINTS", "200"))
```

### `app/agents/viz.py`
При формировании спеки с `type != "none"` заполнять `data` строками из
`state.extraction.rows`, усечёнными до `settings.max_chart_points`. Класть
**полные строки** (все колонки), а не только x/y — для отладки полезно видеть
категориальные и числовые колонки рядом; фронт берёт нужные по `chart.x`/`chart.y`.
Для `type == "none"` (одна строка/одно число) `data` остаётся пустым.

Намеренно **не** чинить здесь маппинг осей — баг должен быть виден на графике.

### `app/agents/answer.py`
Не меняется: уже делает `state.chart.model_dump()`, новое поле подхватится.

### `app/api/main.py`
Новый route `GET /ui` → `FileResponse` со страницей `app/api/static/index.html`
(чтение с диска — правки HTML видны без перезапуска процесса). Корень `/` и
контрактные пути (`/api/chat` и его алиасы) не трогаются.

## Frontend

Один self-contained файл `app/api/static/index.html` (inline CSS/JS, ноль сборки):

- поля ввода: `message` (textarea), `session_id` (input, опц.), кнопка «Спросить»;
- вывод: `response` (текст), бейдж `insufficient_data`, список `assumptions`,
  таблица `trace` (колонки: agent / sql / rows / elapsed_ms), `<canvas>` графика;
- Chart.js подключается из CDN (`cdn.jsdelivr.net`);
- JS делает `fetch('/api/chat', POST, {message, session_id?})` и рендерит ответ.

Маппинг `chart.type` → Chart.js:
| ChartSpec.type | Chart.js |
|---|---|
| `bar` | `bar`: labels = `data.map(r => r[x])`, values = `data.map(r => r[y])` |
| `line` | `line` |
| `area` | `line` с `fill: true` |
| `scatter` | `scatter`: точки `{x: r[x], y: r[y]}` |
| `grouped_bar` | `bar` с группировкой датасетов по `series` (базово) |
| `none` | график не рисуется |

## Обработка ошибок

Страница не должна «падать молча»:
- fetch-ошибка / HTTP 400 / 422 → показать текст ошибки, страница работает;
- `insufficient_data == true`, либо `chart` отсутствует, либо `data` пуст →
  блок «графика нет» + причина из `chart.reason`;
- Chart.js не загрузился (нет внешней сети на VM) → подпись «Chart.js недоступен»,
  остальной вывод (текст/trace/assumptions) работает.

## Тестирование

- **Unit (`viz`):** для bar-спеки `chart.data` заполнено, `data == rows`,
  длина ≤ `max_chart_points`; для `type == "none"` — `data` пусто.
- **Сериализация (`test_api`):** ответ `/api/chat` содержит `chart.data` как
  валидный JSON. Проверить типы из DuckDB (`Decimal`/`date`): если стандартный
  JSON-энкодер их не берёт — приводить значения к сериализуемым в `viz`.
- **Route:** `GET /ui` → 200, `Content-Type: text/html`.
- **Регресс:** существующие 77 оффлайн-тестов зелёные.
- **Ручная live-проверка:** открыть `/ui`, спросить «выручка по линиям» →
  увидеть график и кривую ось Y (наблюдаемость бага).

## Риски

- **CDN недоступен на VM** — график не загрузится, остальное работает. При
  необходимости позже вендорим Chart.js локально в `static/`.
- **Сериализация DuckDB-типов** — может потребовать приведения в `viz`;
  покрыто тестом сериализации.
