# Система оценки, команды, инфраструктура

## ⚠️ Расхождение в системе оценки между источниками

В разных местах хакатона приведены **три разные версии рубрики**. Это надо учитывать и, если можно,
уточнить у организаторов. Все три бьются на 45% авто (LLM-судья) + 55% жюри, но веса разные.

### Вариант A — живой лидерборд (`/`, колонки таблицы) и PDF-брифинг
| Критерий | Вес |
|---|---|
| Точность ответов | 30% |
| Дисциплина границ | 15% |
| Качество интерпретации | 25% |
| Архитектура агентов | 20% |
| Визуализация и UX | 10% |

### Вариант B — страница описания кейса (`/case`, «5-балльная система оценки»)
| Критерий | Вес | Кто |
|---|---|---|
| Точность ответов | 35% | LLM-агент |
| **Экономика решения** | 15% | LLM-агент |
| Архитектура | 10% | эксперт |
| Качество интерпретации | 20% | эксперт |
| Визуализация и UX | 20% | эксперт |

> «Экономика решения» = во сколько обходится один ответ на закрытом наборе: расход токенов, число
> вызовов LLM, длина агентных цепочек, отсутствие холостых итераций критика. Новый критерий, которого
> нет в варианте A. Пайплайн его подтверждает: «оценка по Экономике решения (15%) выше» при работе через план.

### Вариант C — упоминания в тексте пайплайна (`/pipeline`)
`trace` и `assumptions` «работают на оценку «Качество системы» (30%) и «Обоснованность решения» (25%)».
Названия не совпадают ни с A, ни с B — вероятно, устаревшая формулировка.

**Вывод для нас:** проектируем под объединение требований — высокая точность, дисциплина границ
(«данных нет»), бизнес-интерпретация, явные допущения/traceability, экономичность (мало токенов и
итераций), уместная визуализация, чёткая архитектура 4 ролей. Это покрывает любую из рубрик.

**Тестирование (белый хакинг)** — отдельная номинация со своими баллами, вне 100%.

## Состав сдачи
- Работающая система, развёрнутая в **Yandex Cloud**, с прямым доступом жюри.
- Краткое описание архитектуры — схема + 1–2 страницы: какие агенты, как взаимодействуют, какие решения и почему.
- Транскрипт работы агентов хотя бы на одном тест-кейсе — видно, как система пришла к выводу.
- Защита: **7 минут выступление + 5 минут вопросы жюри**.

## Что предоставляют организаторы
- Витрина данных Meridian с легендой (одинакова для всех, не меняется).
- **Открытый набор бизнес-вопросов** для разработки и отладки (закрытый тест-набор — только на оценке).
  Публично на сайте не выложен — вероятно, в личном кабинете команды (логин из письма).
- Технический консультант по записи (помогает с архитектурой агентов, не с интерпретациями).

## Инфраструктура и стек
- **Yandex Cloud** — вычислительные мощности на команду.
- **Claude Code Max** — подписка уже привязана к выданному аккаунту.
- **LLM на выбор** — OpenAI, локальные модели, кастомные пайплайны; модели из Yandex AI Studio.
- **Фреймворки на выбор** — LangGraph, CrewAI, AutoGen или код с нуля.
- Логин/пароль команды едины для SSH / Jupyter / личного кабинета / авто-проверки (из письма).

### Рекомендованный тулинг (из пайплайна)
- **VPN** (часть сервисов недоступна из РФ) — поднять до установки Claude и плагинов.
- **Claude Desktop** — основная среда работы (https://claude.com/download).
- **VS Code** + интеграция Claude Code.
- Плагин **Superpowers** (claude.com/plugins/superpowers).
- Плагин **Context7** — актуальная докуметация библиотек (claude.com/plugins/context7).
- **Chrome DevTools MCP** — отладка дашборда/диалогового интерфейса (github.com/ChromeDevTools/chrome-devtools-mcp).
- Начинать с **Plan mode** в Claude Code: сначала зафиксировать агентов, контракты, формат ответа — потом код. Экономит токены (→ выше балл по «Экономике решения»).
