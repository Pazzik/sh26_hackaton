## ⚠️ Ловушки, которые агент обязан учитывать

### 1. Заказы после даты оттока — 4 403 шт. у 2 651 клиента
- Лаг после `churn_date`: 1–26 дней (медиана 8); статусы в основном `completed` (4 059), остальное cancelled/refunded/disputed.
- Это **не баг** — реалистичный grace-period: договор расторгнут, но уже начатые/оплаченные заказы закрываются.
- **Следствие:** `churn_date` ≠ момент остановки активности. **Нельзя жёстко обрезать активность/GMV по `churn_date`** — потеряешь хвост в ~2 недели. При расчёте «активен ли клиент» опираться на факт заказов за период, а не на дату договора.

### 2. Статус `churned` в `customer_activity_monthly` практически не используется
- Всего **326 строк** `churned` из 608 920; строк activity у ушедших клиентов **после месяца оттока — ноль**.
- После ухода строки активности просто перестают генерироваться (а не помечаются `churned`).
- **Следствие (главный инсайт кейса):** считать отток по `status='churned'` **нельзя** — он почти всегда пустой. Реальный сигнал скрытого оттока — рост `dormant` (61 145) + `churning` (79 020). Нужна метрика **«active economic customer»** (есть заказы/GMV за период), а не `churn_date`/`churned`.

### 3. `signup_date` уходит до окна витрины — 67% клиентов
- 16 812 из 25 000 зарегистрированы **до 2023-01-01**, самый ранний — **2019-01-02**.
- Это корректно: окно транзакций 2023–2025, но клиентская база копилась с 2019.
- **Следствие:** при когортном анализе **нельзя брать `signup_date` как «появился в данных»**. «Новые клиенты периода» ≠ все, у кого `signup_date` в окне. Для новых когорт использовать `unit_economics_monthly.new_customers`.

### 4. 1 037 клиентов (4%) без единого заказа
- Сегменты: SMB 843 / Mid 175 / Large 19; из них **652 уже с `churn_date`** (зарегистрировались и ушли, не заказав).
- **Следствие:** влияет на знаменатели конверсии/retention/ARPU. Их нужно явно исключать или считать отдельной группой, иначе средние занижаются.

### 5. Высокие пропуски — это дизайн, а не мусор
| Поле | Пропуск | Что значит |
|---|---|---|
| `customers.churn_date` | 64.5% (16 127) | **NULL = клиент активен** (по легенде). Не «потерянные данные». |
| `churn_reasons.competitor_named` | 82.7% (7 335) | Конкурента назвали лишь немногие ушедшие. |
| `churn_reasons.nps_at_churn` | 58.0% (5 144) | Не у всех был NPS перед уходом. |
| `churn_reasons.interview_completed` | ~39% = False | **Причины оттока известны лишь у ~61% ушедших.** |

- **Следствие:** причины оттока — **неполная картина**. В ответах про «почему уходят» обязательно фиксировать: «по данным exit-интервью, заполненным у ~61% ушедших». Не выдавать распределение `primary_reason` за полную правду.

### 6. Выбросы в `orders.gmv` — легитимные, не мусор
- 91 экстремальное значение (>3·IQR, порог ≈400k ₽, макс 528 595 ₽) на 681 305 заказов = 0.01%.
- Похоже на крупные реальные сделки, а не на ошибки.
- **Следствие:** для «среднего чека» использовать **медиану / перцентили**, а не `mean` — иначе хвост тянет среднее вверх.

## Шпаргалка: обязательные оговорки в ответах

- Отток → не по `churn_date`/`churned`, а по экономической активности (orders/GMV); упоминать скрытый отток (dormant+churning).
- Активность клиента → не обрезать по `churn_date` (хвост заказов до ~26 дней).
- Новые клиенты / когорты → не по `signup_date` в окне, а через `unit_economics_monthly.new_customers`.
- Причины оттока → «по ~61% заполненных exit-интервью», не полная картина.
- Средний чек / средние суммы → медиана и перцентили, не mean (есть легитимные выбросы).
- Метрики на клиентскую базу → помнить про 1 037 клиентов без заказов в знаменателе.
